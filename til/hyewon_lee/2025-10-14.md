### 🍅 JIRA 프로젝트 관리 방식에 대한 고민

지난 두 번의 프로젝트를 진행하며 Jira를 활용한 프로젝트 관리 방식에 대해 지속적으로 고민해보았음. 특히, 팀의 진행 상황을 시각적으로 보여주는 **번다운 차트**가 실제 진행 속도와는 다르게 프로젝트 막바지에 절벽처럼 급격히 떨어지는 형태로 나타나는 현상을 해결하고 싶었다.

1. **과거의 방식들**

   - **첫 번째 프로젝트**
     - 필요한 기능이 생길 때마다 스토리를 생성하고 개인에게 할당하는 방식으로 진행했다
     - 별도의 하위 작업을 생성하지 않고, 스토리 단위로만 작업을 관리했다
     - 한계: 전체적인 계획을 미리 세우기 어렵고, 진행 상황을 체계적으로 추적하기 어려웠다
   - **두 번째 프로젝트**
     - 프로젝트 시작 전 (아이디어 기획 후), 구현할 모든 기능을 스토리로 정의하고 팀원들과 함께 스토리 포인트를 산정했다
     - 스프린트가 시작되면, 백로그에서 이번 주에 진행할 스토리를 가져왔다
     - “회원가입을 할 수 있다”와 같은 스토리에 `[BE] 회원가입 API 개발`, `[FE 회원가입 UI 구현` 과 같은 구체적인 작업을 연결하여 진행했다.
     - 문제점: 이 방식은 보다 전체 프로젝트 관리에 유용했지만 근본적인 문제 해결엔 도움이 되지 않았음.
       개발 프로세스가 백엔드의 모든 API 개발 → 프론트엔드 UI 전체 개발 → 마지막에 전체 기능 연동 순으로 진행됐었다. 이 때문에 하나의 스토리에 연결된 BE, FE, 연동 작업이 모두 끝나야만 해당 스토리가 완료 처리될 수 있었다.
       결국 번다운 차트는 프로젝트 마감 직전에야 수직으로 하강하는, 예측 도구로서의 의미가 없는 형태를 띠게 되었다.

1. **근본적인 원인 분석: Story와 Task의 이해**

이 문제를 해결하기 위해서 Jira 의 기본 이슈 단위인 에픽, 스토리, 작업의 개념을 다시 정리할 필요가 있었다. [참고 자료](https://sam-repository.tistory.com/16)

- 에픽: 여러 스프린트에 걸친 거대한 기능 단위
- 스토리: 사용자 관점에서 가치를 제공하는 기능 요구사항. 상위 에픽을 구성하는 작은 기능.
- 작업: 스토리를 완성하기 위한 개발자 관점의 구체적인 할일

지난 프로젝트의 문제는, 하나의 스토리를 완료하기 위한 작업(Task)를 병렬적으로 처리하지 않고, 작업 유형(BE, FE) 별로 묶어 수평적으로 처리했다는 점이다.

1. **해결 방안 및 이번 프로젝트 적용 계획**

따라서 이번 프로젝트에서는 두 번째 프로젝트의 사전 계획 방식은 유지하되, 수직적 분할 전략을 도입하고자 한다.

- **스토리 단위의 완성을 지향** : 개발 순서를 기능 레이어(BE, FE)가 아닌, 하나의 완전한 스토리 단위로 변경한다.
- **Task의 동시 진행**: 하나의 스토리를 완성하기 위한 `API 개발`, `UI 구현` , `연동` 등의 Task를 한 스프린트 내에 모두 완료하는 것을 목표로 한다.
- **우하향 번다운 차트 목표:** A 기능 스토리가 완전히 완료되면, B 기능 스토리 개발 목표로 넘어가도록 한다.

이번 프로젝트에서는 단순한 일감 나열 보다는 실제로 프로젝트의 운영에 사용하는 도구로서 Jira를 활용해보고자 한다.
