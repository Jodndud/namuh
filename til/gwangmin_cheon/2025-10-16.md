> 📓 **사전 학습**
> 
> - **프로세스 = 실행 중인 프로그램 전체** (독립적, 무겁다, 안전하다).
> - **스레드 = 프로세스 안의 작은 실행 단위** (가볍다, 빠르다, 위험성 있다).

### ❓ RESTful API란?

- REST는 Representational State Transfer의 약자로, 웹에서 자원을 다루는 아키텍처 스타일
- 자원을 URI(고유 식별자)로 표현하고, 자원에 대한 행위는 HTTP 메서드로 정의한다는 점
- 클라이언트와 서버 간의 통신을 단순하고 일관성 있게 만들어줌

### 👍🏻 장점

- 확장성, 가독성, 유지보수성이 뛰어남
- 무상태(Stateless) 특성을 가져, 서버가 클라이언트 상태를 보관하지 않아 효율적임

### 🤨 고려해야할 점

- 모든 상황에서 RESTful이 완벽한 것은 아니며, 실시간성이 중요한 경우엔 WebSocket 같은 방식이 더 적합할 수 있음

### ✔️ 특징

- JSON이나 XML 같은 포맷으로 데이터를 주고받을 수 있지만, 보통은 JSON을 많이 사용함
    - **❓ JSON을 많이 사용하는 이유**
        
        ## 🧩 1️⃣ REST API에서 JSON과 XML이 모두 가능한 이유
        
        REST API는 단순히 **데이터를 어떤 포맷으로 주고받는지**를 강제하지 않아요.
        
        REST는 **“자원을 HTTP로 주고받는 원칙”**을 정의할 뿐, 포맷은 자유롭습니다.
        
        그래서 서버와 클라이언트는 다음 헤더로 서로 원하는 데이터 형식을 주고받습니다.
        
        ```
        Accept: application/json
        Content-Type: application/json
        
        ```
        
        혹은
        
        ```
        Accept: application/xml
        Content-Type: application/xml
        
        ```
        
        즉, **JSON이든 XML이든 REST 규칙에는 어긋나지 않습니다.**
        
        ---
        
        ## ⚖️ 2️⃣ 그런데 왜 JSON이 압도적으로 많이 쓰일까?
        
        ### ✅ (1) **가볍고 단순하다**
        
        - JSON은 `{}`와 `[]`로 이루어진 **간결한 구조**.
        - XML은 `<태그>`와 `</태그>` 구조라서 **태그 오버헤드가 많음**.
        
        | 비교 | JSON | XML |
        | --- | --- | --- |
        | 예시 | `{"name": "Tom", "age": 25}` | `<person><name>Tom</name><age>25</age></person>` |
        | 데이터 크기 | 작음 | 큼 |
        | 파싱 속도 | 빠름 | 느림 |
        | 가독성 | 높음 | 낮음 |
        
        ---
        
        ### ✅ (2) **자바스크립트와 자연스럽게 호환된다**
        
        - JSON은 **JavaScript Object Notation**의 약자예요.
            
            → 즉, **자바스크립트 객체 문법 그대로** 사용 가능.
            
        - 웹 프론트엔드 대부분이 JS로 작성되므로
            
            브라우저에서 별도 변환 없이 바로 사용 가능.
            
        
        ```jsx
        // 예: JSON 응답을 바로 객체로 변환
        const data = await response.json();
        console.log(data.name); // 바로 접근 가능!
        
        ```
        
        XML은 이런 변환을 위해 **DOMParser 등 추가 처리**가 필요해서 번거롭습니다.
        
        ---
        
        ### ✅ (3) **언어 간 호환성 좋음**
        
        - Python, Java, Go, C#, Kotlin 등 거의 모든 언어에서
            
            `json` 라이브러리를 기본 내장 또는 공식 지원.
            
        - XML은 언어마다 처리 방식이 다르고 복잡합니다.
        
        ---
        
        ### ✅ (4) **REST API의 “경량 통신” 철학과 잘 맞는다**
        
        - REST는 “단순하고 효율적인 웹 통신”을 지향합니다.
        - JSON은 이 철학에 딱 맞는 **경량 구조**를 가지고 있습니다.
        - 반면 XML은 원래 문서 구조를 표현하기 위한 언어여서, REST와는 약간 무겁습니다.
        
        ---
        
        ### ✅ (5) **현대 웹·모바일 환경에 최적화**
        
        - 브라우저, 모바일앱, SPA(Single Page App) 등
            
            대부분이 JSON 기반 통신을 표준으로 사용합니다.
            
        - 심지어 DB도 MongoDB, Firebase처럼 **JSON 기반 문서형 데이터 구조**를 채택하고 있죠.
        
        ---
        
        ## 🧠 정리 요약
        
        | 항목 | JSON | XML |
        | --- | --- | --- |
        | 구조 | 간결하고 짧음 | 태그 많고 복잡 |
        | 파싱 속도 | 빠름 | 느림 |
        | 데이터 크기 | 작음 | 큼 |
        | 브라우저 호환성 | 우수 | 낮음 |
        | REST와의 궁합 | 매우 좋음 | 상대적으로 무거움 |
        
        > ✅ 결론:
        > 
        > 
        > REST API는 JSON, XML 모두 가능하지만,
        > 
        > **가볍고 빠르고 JS 친화적인 JSON**이 사실상 표준처럼 쓰입니다.
        > 
- RESTful API는 자원을 직관적으로 표현하고, HTTP 규칙을 활용해 효율적으로 통신하는 표준화된 방법

### 📢 유사질문

- REST와 RESTful을 비교하는 질문으로 출제된 바 있음

### 🆚 구조적 차이

| 구분 | **REST** | **RESTful** |
| --- | --- | --- |
| **정의** | Representational State Transfer의 약자. 웹에서 자원을 다루는 **설계 원칙(아키텍처 스타일)** | REST 원칙을 실제로 **지켜서 구현한 API** |
| **형태** | 개념, 규칙, 제약조건의 집합 (즉 “이론”) | REST 원칙을 따르는 “구현된 API” (즉 “실행”) |
| **핵심 원칙** | - 자원은 URI로 표현- HTTP 메서드(GET, POST, PUT, DELETE 등)로 자원에 대한 행위 정의- 무상태성(Stateless)- 일관성 있는 인터페이스- 클라이언트/서버 분리 등 | 위의 REST 원칙을 실제 코드와 API 설계에서 준수하는 경우 “RESTful하다”고 표현 |
| **예시** | - “RESTful하게 설계해야 한다”는 원칙- “REST API를 만든다”라는 개념 | - `/users/1` (GET → 사용자 조회, DELETE → 삭제)- `/users` (POST → 생성)처럼 REST 규칙을 잘 지킨 API |
| **비유** | **교통법규(REST)** — 자동차가 지켜야 할 규칙 | **도로 위의 차량(RESTful)** — 실제로 법규를 지키며 운전하는 상태 |

👉 **정리하자면:**

- **REST**는 “규칙(원칙)”이고,
- **RESTful**은 “그 규칙을 잘 지켜서 만든 서비스(API)”입니다.

💡 **면접에서의 모범 답안 예시:**

> “REST는 자원을 URI로 표현하고 HTTP 메서드로 상태를 전송하는 아키텍처 스타일이고, RESTful은 그 원칙을 잘 지켜 구현된 API를 말합니다.”
> 

## 🚫 RESTful하지 않은 API (비RESTful)

```
GET /getUserInfo?id=1
POST /createNewUser
PUT /updateUserInfo?id=1
DELETE /deleteUserById?id=1
```

### ❌ 문제점

- **URI가 동사**로 되어 있음 → `getUserInfo`, `createNewUser`
    
    → REST에서는 자원(Resource)을 **명사**로 표현해야 함.
    
- **일관성이 떨어짐** → URI마다 형식이 제각각.
- **HTTP 메서드의 의미를 무시**하고, URI로 행위를 표현함.

---

## ✅ RESTful한 API (좋은 예시)

```
GET /users           → 사용자 목록 조회
GET /users/1         → id=1인 사용자 조회
POST /users          → 새 사용자 생성
PUT /users/1         → id=1인 사용자 수정
DELETE /users/1      → id=1인 사용자 삭제
```

### 🌟 특징

- **자원(Resource)** = `/users`
- **행위(Method)** = HTTP 메서드(GET, POST, PUT, DELETE)로 표현
- **URI는 명사**, **메서드는 동사 역할**
- 일관되고 예측 가능함 (확장성과 유지보수성 ↑)

### 💡 REST / RESTful 관련 면접 **예상 질문 & 모범 답변 정리**

| ✅ 질문 | 💬 모범 답변 (핵심 요약 포함) |
| --- | --- |
| **1. REST란 무엇인가요?** | REST는 Representational State Transfer의 약자로, **웹에서 자원을 효율적으로 표현하고 전송하기 위한 아키텍처 스타일**입니다. 자원을 URI로 표현하고, HTTP 메서드(GET, POST, PUT, DELETE 등)로 해당 자원에 대한 행위를 정의합니다. |
| **2. RESTful API란 무엇인가요?** | REST의 원칙을 지켜 구현한 API를 말합니다. 즉, 자원을 URI로 명확히 표현하고, HTTP 메서드를 적절히 사용하며, 무상태성(Stateless)을 유지하는 API입니다. |
| **3. REST와 RESTful의 차이점은?** | REST는 “설계 원칙”, RESTful은 “그 원칙을 실제로 구현한 것”입니다. 즉, RESTful한 API는 REST의 제약조건을 잘 지킨 API를 의미합니다. |
| **4. REST의 핵심 제약조건은 무엇인가요?** | ① 클라이언트-서버 구조② 무상태성(Stateless)③ 캐시 가능(Cacheable)④ 계층화된 시스템(Layered System)⑤ 일관된 인터페이스(Uniform Interface)⑥ 코드 온 디맨드(Optional) |
| **5. 무상태성(Stateless)이란 무엇인가요?** | 서버가 클라이언트의 상태를 저장하지 않는 것을 말합니다. 각 요청은 독립적이며, 필요한 정보는 요청마다 포함되어야 합니다. 이로 인해 서버 확장성과 효율성이 높아집니다. |
| **6. RESTful API의 장점은 무엇인가요?** | - 확장성(Scalability) 높음- 클라이언트/서버 분리로 유지보수성 우수- HTTP 표준을 활용해 개발이 간단- JSON 기반으로 가볍고 효율적 |
| **7. RESTful API의 단점은 무엇인가요?** | - 실시간성이 필요한 경우(예: 채팅, 게임)는 비효율적- 요청마다 인증 정보가 필요 → 오버헤드 증가- 트랜잭션 유지 어려움 |
| **8. RESTful하지 않은 설계 예시를 들어보세요.** | `GET /getUserInfo?id=1`처럼 **URI에 동사**가 들어가면 RESTful하지 않습니다. REST에서는 `/users/1`처럼 자원을 **명사로 표현**하고, HTTP 메서드로 행위를 구분해야 합니다. |
| **9. RESTful과 GraphQL의 차이점은 무엇인가요?** | RESTful은 정해진 엔드포인트마다 자원을 요청하지만, GraphQL은 **단일 엔드포인트에서 필요한 데이터만 질의(Query)**할 수 있어 **데이터 낭비를 줄이는 장점**이 있습니다. |
| **10. RESTful API를 직접 설계해본 경험이 있나요?** | (예시 답변) “네, Django와 React를 이용한 웹 프로젝트에서 RESTful API를 직접 설계했습니다. `/api/users`처럼 자원을 명사형 URI로 표현하고, CRUD 동작을 HTTP 메서드로 구분했습니다. Postman을 통해 엔드포인트 테스트도 진행했습니다.” |

## 🚀 핵심 비교: **REST API vs FastAPI**

| 구분 | **REST API** | **FastAPI** |
| --- | --- | --- |
| **정의** | REST 원칙을 따르는 **API 설계 방식 (아키텍처 스타일)** | Python 기반의 **웹 프레임워크(도구)** — RESTful API를 쉽고 빠르게 구현하게 해줌 |
| **역할** | "API를 **어떻게 설계할지**"에 대한 개념 | "API를 **어떻게 구현할지**"를 위한 기술/라이브러리 |
| **형태** | 이론, 원칙, 디자인 철학 | 프레임워크 (Python 라이브러리) |
| **언어** | 특정 언어와 무관함 (Java, Node.js, Django 등 어디서든 구현 가능) | Python 전용 |
| **핵심 특징** | - 자원(URI) 중심 설계- HTTP 메서드 사용- 무상태성(Stateless)- 클라이언트-서버 분리 | - Python 타입 힌트 기반 자동 문서화- **Swagger UI** 자동 생성- 비동기(Async) 지원- 높은 성능(Starlette + Pydantic) |
| **사용 목적** | 웹에서 클라이언트-서버 간 데이터를 주고받기 위한 설계 규칙 | RESTful API를 빠르고 간결하게 개발하기 위한 프레임워크 |
| **예시 표현** | `GET /users/1` → id=1 사용자 조회 | FastAPI 코드로 `@app.get("/users/{id}")` 로 구현 |
| **결론적으로** | "REST는 **규칙**"이다. | "FastAPI는 **그 규칙을 쉽게 구현하는 도구**"다. |
